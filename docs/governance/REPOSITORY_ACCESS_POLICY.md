# DETERMA Repository Access Policy

## Purpose

This document defines the canonical repository access policy for DETERMA.

The objective is to:

- preserve knowledge boundaries
- reduce accidental exposure
- formalize repository roles
- standardize access decisions
- support governed technical diligence

---

# Canonical Repository Roles

| Repository Type | Purpose | Default Visibility |
|---|---|---|
| Public Showcase | Awareness, onboarding, positioning | Public |
| Curated Diligence | Guided investor and partner review | Private |
| Internal Core | Canonical implementation and governance memory | Private |
| Restricted Archive | Invention-sensitive and moat-critical material | Restricted |

---

# Access Levels

| Role | Recommended Access |
|---|---|
| Public Visitor | Public Showcase Only |
| Design Partner | Curated Diligence |
| Investor | Curated Diligence |
| Strategic Reviewer | Curated Diligence + Guided Review |
| Core Engineer | Internal Core |
| Founder | Full Access |

---

# Public Repository Rules

Public repositories may contain:

- onboarding material
- public-safe architecture
- sanitized runtime explanations
- public demos
- public presentations
- design partner entry points

Public repositories must not contain:

- patent-sensitive material
- deep governance internals
- replay invalidation semantics
- cryptographic internals
- moat-critical implementation mechanics

---

# Diligence Repository Rules

Diligence repositories should contain:

- sanitized architecture
- selected runtime proof artifacts
- selected governance philosophy
- runtime legitimacy framing
- selected threat scenarios

Diligence repositories should not contain unrestricted internal archives.

---

# Internal Repository Rules

Internal repositories preserve:

- canonical memory
- implementation continuity
- governance internals
- operational doctrine
- strategic sequencing
- invention-sensitive material

---

# Access Governance Rules

## Rule 1

Default to minimum necessary exposure.

---

## Rule 2

Repository access does not imply unrestricted disclosure.

---

## Rule 3

NotebookLM access should follow the same disclosure doctrine as repository access.

---

## Rule 4

When uncertain:

```text
Do not expose moat mechanics through convenience.
```

---

# Canonical Principle

```text
Access should follow trust boundaries, not organizational convenience.
```
