# Execution Authority Is Not Intelligence

Intelligence and authority are fundamentally different system properties.

A model can produce high-quality reasoning and still lack legitimate mutation authority. Reasoning quality describes proposal quality. Authority describes whether mutation is permissible now.

## Capability Does Not Imply Legitimacy

Model capability can optimize planning, synthesis, and prediction. None of those properties establish runtime-bound permission to mutate external systems.

## Why Reasoning Cannot Replace Authority Boundaries

Reasoning is inferential. Authority is allocative.

Reasoning asks: "What should happen?"
Authority asks: "Who may cause it to happen under current conditions?"

These must remain separate.

## Authority Outside the Model

Execution authority should exist outside model cognition so that legitimacy checks are not self-issued by the same system proposing mutation.

This preserves:
- independent control boundaries
- runtime-time legitimacy checks
- explicit separation of proposing and mutating

## Distributed Systems Analogy

In distributed systems, consensus correctness is distinct from node intelligence. A fast node does not gain unilateral commit authority. Similarly, a capable model does not gain autonomous mutation authority.

## Operational Analogy

An engineer may draft an excellent migration plan. Production mutation still requires execution authority under current operational constraints.

## Mutation Consequences

Mutation is state-changing, not descriptive. Once executed, rollback may recover function but cannot erase temporal side effects, propagation windows, or trust impact.

## Bounded Execution Philosophy

A robust system allows intelligence to propose broadly and authority to execute narrowly. This pairing enables progress without collapsing legitimacy boundaries.
