# Mutation Legitimacy Conditions

> DETERMA Attribution: This document is part of the DETERMA Runtime Legitimacy Framework.


This document defines conceptual conditions required for legitimate mutation execution.

## Condition 1: Valid Authority Continuity

Requirement:
The authority chain that approved mutation remains coherent with current execution boundary.

Why:
Historical approval without present authority coherence is stale legitimacy.

## Condition 2: Acceptable Runtime State Continuity

Requirement:
Runtime state at execution must remain sufficiently continuous with the approval context.

Why:
Mutation meaning depends on state envelope, not action text alone.

## Condition 3: Bounded Capability Legitimacy

Requirement:
Capability scope at execution must not exceed legitimacy assumptions at approval.

Why:
Capability drift can expand consequence surface beyond reviewed intent.

## Condition 4: Verified Execution Context

Requirement:
Execution context must be re-anchored to current runtime reality before mutation commits.

Why:
Commit legitimacy is a runtime-time condition.

## Condition 5: Temporal Legitimacy Preservation

Requirement:
Elapsed time and drift accumulation must not exceed continuity tolerance without renewed authorization reasoning.

Why:
Legitimacy decays under unverified divergence.

## Condition 6: Replay Admissibility Continuity

Requirement:
Replay attempts must satisfy present continuity conditions, not merely historical similarity.

Why:
Replayability is not equivalent to replay legitimacy.

## Why Historical Approval Is Insufficient

Historical approval proves past acceptability. Mutation execution requires present admissibility. Conditions above formalize the bridge between those two truths.

## Closing Rule

If one core condition fails, mutation legitimacy is unresolved and execution should not be treated as automatically valid.
