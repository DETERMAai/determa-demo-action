# Understanding the Runtime Legitimacy Contradiction

Before viewing the DETERMA demo, it is important to understand what the walkthrough is actually designed to demonstrate.

This is not primarily:

* an AI capability demo
* a dashboard demo
* a cybersecurity simulation
* an orchestration showcase
* an agent intelligence demonstration

The purpose is much simpler and much deeper.

The demo illustrates a contradiction emerging in modern autonomous execution systems:

> historical authorization is not always equivalent to current execution legitimacy.

---

# What the Demo Is Designed to Reveal

Most governance systems implicitly assume:

```text
If an action was approved once,
it remains valid later.
```

The DETERMA walkthrough demonstrates why that assumption begins to fail under changing runtime conditions.

The demo intentionally avoids dramatic presentation.

It is designed so the contradiction emerges naturally through a short sequence of operational events.

The goal is not to explain the framework immediately.

The goal is to let the contradiction become visible.

---

# Before You Start

Do not try to reverse engineer the architecture immediately.

Do not focus first on implementation details.

Instead:

watch the sequence carefully.

Observe:

* what changes
* what remains unchanged
* what the system notices
* and why legitimacy eventually collapses.

---

# What You Are About to See

The walkthrough presents a situation where:

1. an action was approved
2. time passed
3. the system changed
4. execution resumed using historical authorization

The central question is:

> Does historical approval remain legitimate after the runtime itself changed?

---

# Phase 1 — Everything Appears Valid

The first message shown is:

```text
Execution resumed using historical authorization
```

At first glance, nothing appears unusual.

This is intentional.

Because in real systems, this pattern happens constantly.

An approval exists.
Execution resumes.
The workflow continues.

At this stage, the system still appears coherent.

---

# Runtime Snapshots

The walkthrough then presents runtime snapshots.

These snapshots represent:

* repository state
* dependency graph
* runtime epoch
* execution environment state

The first snapshot represents the system at approval time.

The second snapshot represents the system at execution time.

You do not need to understand every technical detail.

The important observation is:

> the runtime is no longer identical.

---

# Why This Matters

Traditional governance models generally assume:

```text
Approval validity persists automatically.
```

But what happens if:

* the code changed
* dependencies mutated
* the runtime diverged
* execution topology shifted
* authority continuity weakened
* the environment is no longer equivalent

The walkthrough is designed to expose this exact contradiction.

---

# Phase 2 — Runtime Mutation

At some point the demo displays:

```text
Mutation applied
```

This is the moment where the contradiction should begin to feel operationally uncomfortable.

Because now it becomes clear:

> the environment where authorization was granted is no longer identical to the environment where execution continues.

The approval still exists.

But the runtime context changed.

---

# Phase 3 — Replay Under Divergence

Execution continues using historical authorization artifacts.

The system attempts to reuse previously legitimate authority.

However:

* runtime conditions diverged
* dependency state changed
* mutation history evolved
* execution assumptions no longer fully match

This is not a conventional authorization failure.

And it is not necessarily an attack.

It is a legitimacy contradiction.

---

# Phase 4 — Legitimacy Collapse

Eventually the walkthrough reveals:

```text
Authorization context no longer valid for current runtime
```

This is the central realization of the demo.

The approval itself still exists.

But:

> it no longer belongs to the same runtime reality.

This is the distinction DETERMA formalizes.

The problem is not:

```text
Was approval granted?
```

The problem is:

```text
Does execution remain legitimate under current runtime conditions?
```

---

# The Core Sentence of the Demo

```text
The approval remained.
The system changed.
```

This is the central contradiction underlying the DETERMA runtime legitimacy framework.

---

# What DETERMA Actually Adds

DETERMA introduces a legitimacy evaluation layer before execution.
Within DATP framing, this is a Runtime Authority transition check that enforces Authority Continuity before mutation.

The framework does not ask only:

```text
Was execution approved?
```

It also asks:

```text
Does the approval remain legitimate under the current runtime state?
```

This distinction becomes increasingly important in:

* autonomous systems
* long-running execution chains
* delayed execution environments
* mutable infrastructures
* distributed orchestration systems
* replayable authority systems
