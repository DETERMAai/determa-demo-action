Why DETERMA Exists

A compressed explanation of the runtime legitimacy contradiction.

The approval remained. The runtime changed. Execution legitimacy diverged.

Runtime Flow

Approval Granted ↓ Runtime Mutated ↓ Replay Attempt ↓ Legitimacy Diverged ↓ Execution Decision

Traditional systems only preserve approval history. DETERMA checks whether that approval still belongs to the current runtime state before execution continues.

What Problem Emerges

Autonomous systems execute after delay, under changing dependencies, evolving repositories, and shifting runtime conditions. Approval artifacts survive, but runtime reality does not stay still.

Why Existing Systems Fail

  • They validate that approval existed, not that runtime continuity still holds.
  • They allow replay under mutated context if credentials or artifacts still look valid.
  • They rely on monitoring after mutation rather than legitimacy checks before mutation.

Contradiction Example

At approval time, runtime snapshot = A17F / D44C / epoch 1021. At execution time, runtime snapshot = B92K / E11X / epoch 1048. Approval record is unchanged, but execution context is not. Historical validity survives. Runtime legitimacy does not.

Why Now

Machine-speed automation increased delay-tolerant execution, dependency churn, and replayable authority paths. These conditions make static approval models structurally incomplete. Runtime legitimacy is becoming an infrastructure requirement, not a policy preference.
